Why DevAgent exists

Dependency bots are safe but shallow. LLM agents are capable but terrifying. Pick neither.

Hosted bots stop at the version bump — when the build breaks, a human cleans up. General-purpose LLM agents will happily fix the build, but they want a shell, your credentials and the benefit of the doubt.

DevAgent splits the difference with an architecture instead of a compromise: a deterministic pipeline does the predictable work, and a strictly caged AI is woken only when something breaks — inside a throwaway container, holding seven tools and nothing else.

The first agent, DependencyPilot, watches your NuGet feed and turns every new package version into a tested, reviewable pull request.

The security model

Six invariants. Enforced in code, locked by tests.

Not a policy PDF — a type system. Each rule below is structural: the API surface that would violate it does not exist, and a test fails if anyone adds it.

INV-01

The AI never touches a shell

The agent acts through seven structured, typed tools. There is no bash, no exec, no “run command” — not forbidden by prompt, but unreachable by construction.

ENFORCED BY
Forge.ToolPolicy · closed tool-call type set
test: Tool_surface_is_exactly_the_seven_allowed_tools
INV-02

Everything is allowlisted

Repositories, packages, container images, job types and commands resolve from administrator allowlists — by key. The request DTOs have no field for a raw URL, an image name or a Docker argument.

ENFORCED BY
Guard.RepositoryPolicy · PackagePolicy · JobPolicy
test: NoUserSuppliedInfrastructureTests
INV-03

The output is a pull request, never a merge

Branch protection and human review stay mandatory. No auto-merge code path exists anywhere in the platform, and the placeholder Git provider throws if anyone asks for one.

ENFORCED BY
Bridge.Git.IGitProvider · AutoMerge rejected
test: Auto_merge_request_is_refused
INV-04

Work happens in a throwaway sandbox

Each job runs in a fresh container launched with --rm, --cap-drop=ALL, no-new-privileges, CPU/memory/pid limits, no volume mounts and no Docker socket. The workspace dies with the container.

ENFORCED BY
Runner.CliSandboxJobRunner · fixed arg vector
test: No_mounts_sockets_privileges_or_ports_ever
INV-05

Every decision is evidence

Allow/deny decisions, prompts, every tool call and every generated diff land in an append-only audit log. When someone asks “what did the AI do?”, you answer with records, not recollection.

ENFORCED BY
Audit.IAuditLog · ToolCall/Diff/Prompt/Decision events
test: Agent_reads_patches_and_rebuilds_then_completes
INV-06

Deterministic first, AI second

The update itself is plain XML editing plus dotnet restore/build/test. The agent wakes only on failure, only if opted in — and its work is re-verified deterministically before anyone trusts it.

ENFORCED BY
Worker.NuGetUpdateWorker · re-verify after repair
test: Agent_success_claim_is_not_trusted_without_green_reverify

How it works

Four stages. One direction. No shortcuts.

A new package version — from the hourly feed check or a webhook — flows through the platform in one direction. Nothing skips the gate, and nothing merges itself.

The AI layer — DevAgent.Forge

A coding agent with seven tools and no exit.

When a dependency bump breaks the build, the agent gets exactly enough capability to fix code — and structurally nothing else. Tool calls are a closed set of types: a tool that isn’t defined can’t be called, hallucinated or smuggled in.

The agent can

list_files read_file apply_patch replace_file run_dotnet_build run_dotnet_test git_status

Each call is path-validated against the workspace, checked against the protected-file policy, and written to the audit log before the result returns to the model.

The agent cannot

bash powershell curl wget ssh docker kubectl az aws exec

Also unreachable: secret files (never readable, never writable), deployment files (locked unless policy says otherwise), any path outside the repository checkout, and any package or repo not on the allowlist.

The paper trail

“What did the AI do?” has an exact answer.

Every policy decision, every prompt, every tool call and every generated diff is an append-only audit event, correlated by job id. Reviews, incident response and compliance all read from the same record.

The agent’s final diff and reasoning summary are saved with the job — so the pull request explains itself, and six months later you can still reconstruct exactly what happened and why.

ToolCallAuditEvent · job f3a91c
{
  "toolName": "apply_patch",
  "arguments": "path='src/Logging.cs' patchBytes=412",
  "allowed": true,
  "actor": "CodingAgent"
}
DecisionAuditEvent · job 8c2e40
{
  "decision": "validate-repository",
  "allowed": false,
  "reason": "Repository 'prod-secrets' is not on the allowlist."
}

One platform, many agents

Two agents shipping. The rails are the product.

Every agent — including yours — inherits the same gate, the same sandbox, the same audit trail. Write the agent, not the security model.

● shipping

DependencyPilot

Watches your NuGet feed, finds affected repositories, bumps the version in a sandbox, builds, tests, repairs on failure — and opens the PR.

planned

PipelineDoctor

Reads failed CI pipelines, diagnoses the break and proposes the fix as a reviewable change.

planned

First-pass review on incoming PRs: policy-aware, audit-logged, and never the final word.

planned

SplunkSentinel

Turns recurring log anomalies into tickets with a suspected root cause attached.

planned

ConfluenceGuide

Keeps runbooks and architecture docs in sync with what the code actually does.

● shipping

.NET Upgrader

Nightly sweep that upgrades every project in watched repositories to a new target framework — built, tested, repaired if needed, and PR’d.

Provider-agnostic by construction: GitHub, GitLab, Azure DevOps and Bitbucket sit behind one Git abstraction; rootless Podman (or Docker) today, Kubernetes Jobs tomorrow behind one sandbox interface; Claude, GPT and Gemini clients ship in the box behind one LLM interface — each agent can pin its own model, or run with none.

Run it locally

From clone to first gated job in three commands.

STEP 1

Start the platform

Hub (dashboard + API + schedule) and Runner (the gate) come up in stub sandbox mode — safe on any laptop.

docker compose up --build
STEP 2

Trigger a job

Ask DependencyPilot for an update. Keys, not URLs — the Runner resolves them against its allowlists.

curl -X POST localhost:5080/hub/dependencypilot/nuget-update \
  -H "Content-Type: application/json" \
  -d '{"repositoryKey":"sample-service",
       "packageId":"Serilog",
       "targetVersion":"3.1.1"}'
STEP 3

Watch the dashboard

Open the live agent-status dashboard, or poll the API; try a repository that isn’t allowlisted and watch it bounce.

open http://localhost:5080/
curl localhost:5080/jobs
# every job, every status — every decision logged
