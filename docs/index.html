.NET 8 · Politik-styret skelet · LLM-klienter: Claude · ChatGPT · Gemini

DevAgent

En sikker platform til at køre interne AI-agenter for udvikler-workflows. Første agent — DependencyPilot — opdager nye NuGet-versioner og åbner pull requests med opdateringer. Hvert job løber gennem en politik-styret valideringsgate, før noget som helst eksekveres.

01Overblik

Hvad systemet er, og hvad det ikke er (endnu).

🎯 Formål

Køre interne, sikre udvikler-agenter — planlagt, event-baseret og senere interaktivt. Resultatet er altid en pull request, aldrig en auto-merge.

🤖 DependencyPilot

Den første agent: opdager nye NuGet-versioner, finder berørte repos og åbner PR'er, der opdaterer afhængigheder sikkert.

🚧 Status

Et deterministisk, politik-styret skelet med LLM-klienter til Claude (standard), ChatGPT og Gemini. Hver agent kan vælge sin egen model, og modellen kan kun handle via strukturerede, politik-checkede værktøjer.

02Arkitektur

To webservices + en sandbox-worker. Hver pil skærer adgang væk.

request-flow
  Hub.Api  ──(nøgler + version)──▶  Runner.Api  ──(valideret, opslået)──▶  Sandbox-worker
     │                              │                                  │
  manuel trigger /            allowlist-gate:                  clone ▸ branch ▸
  Hangfire-tidsplan           repo? pakke? image? jobtype?     opdatér PackageReference ▸
                              opslår NØGLE ▸ betroet URL/image  restore/build/test ▸
                                                                push ▸ åbn PR (aldrig merge)

Hub.Api · port 5080

Hoveddøren. Tager manuelle triggers og kører Hangfire-tidsplanen. Videresender til Runner over HTTP — sender kun nøgler, aldrig URL'er.

Runner.Api · port 5081

Valideringsgaten. Re-validerer hvert job mod allowlists for repo / pakke / image / jobtype og opslår nøgler til betroede URL'er og images.

03Kom i gang

Forudsætning: .NET 8 SDK. Faste porte er sat op via launchSettings.json.

Byg og kør testene

bash
dotnet build DevAgent.sln
dotnet test  DevAgent.sln   # 134 tests skal bestå

Start Runner (valideringsgaten) — denne skal køre først

terminal 1
dotnet run --project src/DevAgent.Runner.Api --launch-profile Runner

Lytter på http://localhost:5081 · åbner /health i browseren.

Start Hub (i et andet vindue)

terminal 2
dotnet run --project src/DevAgent.Hub.Api --launch-profile Hub

Lytter på http://localhost:5080 · åbner Hangfire-dashboardet automatisk.

Brug systemet

Åbn dashboardet på localhost:5080/hangfire eller trigger et job via API — se eksemplerne nedenfor.

Tip: Med launchSettings.json på plads behøver du ikke sætte ASPNETCORE_URLS — F5 i Visual Studio / Rider eller dotnet run --launch-profile bruger nu de faste porte 5080 og 5081.

04Hvor er brugergrænsefladen?

Der er bevidst ingen skræddersyet app-UI endnu — systemet styres via HTTP-API'er.

📊 Hangfire-dashboard

Den eneste indbyggede webside: /hangfire på Hub'en. Her ser du planlagte, kørende og fejlede jobs — fx det timebaserede dependencypilot-package-check.

📡 REST-API

Al styring sker via JSON over HTTP — fra curl, Postman, et script eller (på sigt) en frontend. Denne side er din interaktive reference.

Denne HTML-side er dokumentation, ikke en live-konsol. Kør kommandoerne i din terminal — knapperne kopierer dem til udklipsholderen.

05API-eksempler

Kopiér-klar curl. Bemærk: repositoryKey og packageId er allowlist-nøgler, ikke rå URL'er.

GET  Health-check

begge services svarer ens
curl http://localhost:5080/health   # Hub
curl http://localhost:5081/health   # Runner

# ▸ {"status":"ok","service":"DevAgent.Hub"}

POST  Trigger et NuGet-update via Hub (den normale vej)

POST /hub/dependencypilot/nuget-update
curl -X POST http://localhost:5080/hub/dependencypilot/nuget-update \
  -H "Content-Type: application/json" \
  -d '{
    "repositoryKey": "sample-service",
    "packageId":     "Serilog",
    "targetVersion": "3.1.1",
    "requestedBy":   "nikolaj"
  }'

Hub'en videresender til Runner på localhost:5081. Kører Runner ikke, får du HTTP 500 — start Runner først.

POST  Kald Runner direkte — accepteret job

POST /runner/jobs/nuget-update · ✓ allowlisted
curl -X POST http://localhost:5081/runner/jobs/nuget-update \
  -H "Content-Type: application/json" \
  -d '{"repositoryKey":"sample-service","packageId":"Serilog","targetVersion":"3.1.1"}'

# ▸ HTTP 200
# {"jobId":"d11f…","status":2,   ◂ Validated
#  "message":"Podman sandbox runner is a stub in the first milestone; no container was started."}

POST  Kald Runner direkte — afvist af politik

POST /runner/jobs/nuget-update · ✗ ikke på allowlist
curl -X POST http://localhost:5081/runner/jobs/nuget-update \
  -H "Content-Type: application/json" \
  -d '{"repositoryKey":"sample-service","packageId":"EvilPackage","targetVersion":"1.0.0"}'

# ▸ HTTP 400
# {"jobId":"3ce0…","status":7,   ◂ Rejected
#  "message":"Package 'EvilPackage' is not on the allowlist."}
Dette er kernen i sikkerhedsmodellen i praksis: et job med en ikke-godkendt pakke afvises før eksekvering med en klar besked og status 7 (Rejected).

06Job-statuskoder

Feltet status i svaret (AgentJobStatus).

KodeNavnBetydning
0UnknownUkendt / ikke sat.
1PendingAccepteret af Hub, endnu ikke valideret/afsendt.
2ValidatedBestod Runner-validering; en worker forberedes.
3RunningEn sandbox-worker eksekverer jobbet.
4SucceededFuldført — fx en pull request blev åbnet.
5NoChangeFuldført uden ændring (fx allerede opdateret).
6FailedWorker kørte, men build/test fejlede.
7RejectedAfvist før eksekvering (politik-/allowlist-brud).

07Allowlist-config

Alle godkendte repos, pakker og images bor i src/DevAgent.Runner.Api/appsettings.json under Guard. Kun administratorer redigerer den.

appsettings.json (uddrag)
"Guard": {
  "Repositories": [
    { "Key": "sample-service",
      "CloneUrl": "https://git.internal.example.com/platform/sample-service.git",
      "BaseBranch": "main" }
  ],
  "Packages": [ "Serilog", "Newtonsoft.Json", "Polly" ],
  "ContainerImages": [ "registry.internal.example.com/devagent/worker-dotnet:8.0" ],
  "JobTypeImages": {
    "NuGetUpdate": "registry.internal.example.com/devagent/worker-dotnet:8.0"
  }
}

Vil du tillade en ny pakke eller et nyt repo, så tilføj nøglen her — callere refererer altid kun til nøglen, aldrig URL'en eller image-navnet.

08Sikkerhedsmodel

Håndhævet, ikke aspirativt.

🔒 Ingen rå adgang

LLM'en får aldrig host-shell, Podman/Docker-socket, SSH, secrets eller deploy-/cloud-credentials.

🔑 Kun nøgler

Callere sender allowlist-nøgler — aldrig rå URL'er, images eller container-argumenter.

🚪 Re-validering

Runner re-validerer hvert job mod repo/pakke/image/jobtype-allowlists.

⛔ Ingen shell

SafeCommandRunner kører kun dotnet og git som argument-vektor — ; && | $() er inerte.

🧭 Sti-vagt

WorkspacePathValidator blokerer path-traversal og absolutte sti-escapes; ProtectedFilePolicy blokerer ændringer i secrets/deploy-filer.

🔁 Altid en PR

Resultatet er altid en pull request — der findes ingen auto-merge-kodevej nogen steder. Alt auditeres.

Den fremtidige LLM-agent (DevAgent.Forge) kan kun handle gennem syv strukturerede værktøjer — list_files, read_file, apply_patch, replace_file, run_dotnet_build, run_dotnet_test, git_status — sti-valideret, allowlist-checket og auditeret. Ingen shell, curl, ssh eller docker.

09LLM-providere & model pr. agent

Forge-agenten kan drives af Claude, ChatGPT eller Gemini — og hver agent vælger sin egen model.

Claude · standard

Anthropic Messages API. Standardmodel claude-opus-4-8. Nøgle: ANTHROPIC_API_KEY.

ChatGPT

OpenAI Chat Completions. Standardmodel gpt-4o. Nøgle: OPENAI_API_KEY.

Gemini

Google generateContent. Standardmodel gemini-2.0-flash. Nøgle: GEMINI_API_KEY.

Alle tre implementerer det samme ILlmClient og oversætter agentens syv strukturerede værktøjer til provider-værktøjskald. Modellen kan aldrig kalde en shell — kun de allowlist-checkede værktøjer.

Vælg provider og model for en agent

En agent angiver sin provider og model i konfigurationen (her DependencyPilot's Llm-sektion):

appsettings.json
"DependencyPilot": {
  "Llm": {
    "Provider": "Claude",        // Claude | OpenAi | Gemini
    "Model": "claude-opus-4-8"  // udelad for provider-standard
  }
}
C# — byg klienten
var options = new LlmClientOptions { Provider = LlmProvider.Gemini, Model = "gemini-2.0-flash" };
var llm = LlmClientFactory.Create(options, httpClient);
var agent = CodingAgentFactory.Create(workspace, llm, audit, jobId);
Nøglen læses fra miljøet (eller LlmClientOptions.ApiKey). Mangler den, fejler kaldet højlydt med en klar besked — aldrig stille degradering.

Build-repair i sandbox-worker'en

Forge-agenten er nu wiret ind i worker'en (RepoWorkflow) som et opt-in build-repair-trin: hvis bygningen fejler efter en ændring, og DEVAGENT_LLM_PROVIDER er sat, får agenten ét afgrænset forsøg på at rette fejlen via sine værktøjer. Derefter verificerer worker'en deterministisk igen og åbner en review-krævende PR — agenten merger aldrig.

Ny agent: .NET-opgradering

Agents.DotNetUpgrader foreslår at opgradere target frameworket (fx net6.0 → net8.0) på alle projekter i et repo. Worker'en udfører det deterministisk via TargetFrameworkUpdater (rører kun moderne net<x>.<y>, downgrader aldrig, lader netstandard/net4x være) — og framework-bumps der bryder bygningen er netop hvor build-repair-trinet ovenfor hjælper. Worker'en vælger job via DEVAGENT_JOB_TYPE (NuGetUpdate | DotNetUpgrade).

10Nuværende begrænsninger

Det er en første milepæl — vid hvad der er ægte og hvad der er stub.

Stub-eksekvering: PodmanSandboxJobRunner samt git-/NuGet-providerne er placeholders. Et accepteret job når valideringsgaten og returnerer Validated, men der startes ingen container, og der oprettes ingen rigtig pull request endnu.
LLM kræver en API-nøgle: Klienterne til Claude/ChatGPT/Gemini er på plads (DevAgent.Bridge.Llm), men kører kun når den relevante nøgle er sat i miljøet (ANTHROPIC_API_KEY / OPENAI_API_KEY / GEMINI_API_KEY). Build-repair-trinet i sandbox-worker'en er stadig opt-in.
Kun lokal sikring: Hangfire-dashboardet på /hangfire er åbent lokalt — sikr det ordentligt før eksponering uden for din maskine.

Du kan altså i praksis afprøve hele API-flowet og politik-valideringen nu — det er netop dét, skelettet er bygget til at demonstrere sikkert.
